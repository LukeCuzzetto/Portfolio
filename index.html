03 / About

How I work.

I like understanding what a system needs to do, modelling it clearly, and building it in small pieces that can be tested and explained. My goal is not to collect technologies, it is to become dependable at solving backend problems.

  • Understand before abstracting. Start with the real domain and introduce complexity only when it earns its place.
  • Build complete slices. Prefer a small feature that is validated, stored, tested, and documented over several half-built layers.
  • Communicate clearly. Good engineering includes explaining decisions, surfacing uncertainty, and leaving useful context for others.
  • Own the result. Follow work through testing, delivery, feedback, and maintenance rather than stopping when the code compiles.
